# API Reverse Engineering Toolkit

## Project Overview

An interactive API reverse engineering toolkit for Claude Code. The user captures web traffic as HAR (HTTP Archive) files — either exported from browser DevTools or generated by mitmproxy — then the toolkit analyzes those captures to extract metadata about target pages and guide users through a structured process of discovering, mapping, and documenting APIs — ultimately producing a complete API definition that can be used for client code generation.

The toolkit turns API reverse engineering from a manual, ad-hoc process into a guided, conversational investigation.

## Core Principles

- **User defines scope, capture provides everything else.** The user tells us the target and what data they care about. Then they browse the site. All analysis, metadata, endpoint discovery, and reporting is derived from the captured traffic — not from guessing or asking the user to perform specific actions.
- **Multi-session intelligence.** Single requests show shape. Multiple requests across sessions reveal behavior. Always gather enough samples to understand what's static vs dynamic.
- **Honest feasibility assessment.** If anti-scraping measures make programmatic access impractical, say so early — before the user invests time in later phases.
- **The API definition is the primary artifact.** Code generation is derived from it. The definition is what we're building toward.

## Phase 1 — RECON

### Goal

Build a comprehensive context profile of the target page before examining specific endpoints. This profile informs every subsequent phase.

### User Input

- **Target page:** The URL the user wants to reverse-engineer (e.g., `tipsport.cz`)
- **Data of interest:** What the user actually wants to extract (e.g., events, odds, scores)

Note: The API may live on a completely different domain than the target page. A betting site's frontend might pull data from third-party odds providers, CDN feeds, or separate API subdomains. Never assume the API domain matches the target page.

### What We Gather

#### API Surface

- All domains the page communicates with
- Classification of each domain: data API vs static assets vs analytics/tracking
- Base URL patterns (e.g., `api.example.com/rest/v1/...`, `api.example.com/graphql`)
- API style: REST, GraphQL, RPC, or a mix

#### Frontend Stack

- Framework: React, Angular, Vue, Svelte, vanilla JS
- HTTP client library: axios, fetch, custom wrapper
- State management layer: Redux, MobX, Vuex, signals — may hint at data structures
- Any bundled API client or SDK in the JavaScript

#### Real-time Layer

- Active WebSocket connections — to which domains
- WebSocket library: native WebSocket, Socket.io, SignalR, SockJS
- Protocol over the WebSocket: raw JSON, protobuf, custom framing
- Server-Sent Events (SSE) presence alongside or instead of WebSockets

#### Auth & Session

- Auth mechanism: cookies, bearer tokens, API keys in headers or query params
- Login wall presence — does public data flow without authentication?
- Tokens embedded in page source or initial JS payload
- OAuth flows, CSRF tokens

#### Protection Layer

- CDN/WAF: Cloudflare, Akamai, Imperva, Fastly
- CAPTCHA challenges: reCAPTCHA, hCaptcha, Cloudflare Turnstile
- Anti-bot headers or browser fingerprinting scripts
- Request signing: are API calls signed with rotating or JS-derived tokens?
- Path or payload obfuscation

### User Interaction During Recon

1. **User defines scope.** User provides the target URL and describes what data they're interested in (e.g., "I want events, odds, and scores from tipsport.cz").
2. **User browses the site.** The user navigates the site naturally — visiting pages that contain the data they care about. The user decides where to go and what to click. Traffic is captured via one of two methods:
   - **Browser DevTools:** Open the Network tab before browsing.
   - **mitmproxy:** Run mitmproxy as an intercepting proxy — captures traffic from any HTTP client, not just browsers.
3. **User exports HAR.** When the user feels they've covered the areas they're interested in, they export the capture as a HAR file and provide it to the toolkit:
   - **From DevTools:** Right-click → "Save all as HAR with content" (Chrome), or equivalent in other browsers.
   - **From mitmproxy:** Use `mitmdump --set hardump=./capture.har` during capture, or export from mitmweb's UI.

### HAR File Handling

- HAR files contain full request/response pairs including headers, query parameters, request bodies, response bodies, cookies, and timing data.
- A single HAR file represents one browsing session. Multiple HAR files across sessions enable differential analysis.
- HAR files may be large — the toolkit should handle them efficiently, filtering out irrelevant entries (e.g., static assets, analytics) early.
- Response bodies in HAR files may be base64-encoded — decode before analysis.
- WebSocket frames are captured in HAR under `_webSocketMessages` (Chrome extension) or may require separate capture depending on the browser.
- mitmproxy captures WebSocket frames natively and includes them in its HAR export, making it the better choice when real-time data is important.

### What Claude Does With the Capture

All analysis happens after the user provides the HAR file — entirely derived from the captured traffic:

- **Domain classification.** Categorize every domain contacted: which serve data APIs, which are static assets, which are analytics/tracking.
- **Stack detection.** Identify the frontend framework, HTTP client library, and state management from the JavaScript bundles and request patterns.
- **Real-time discovery.** Detect WebSocket or SSE connections, identify the library and protocol from the handshake and message patterns.
- **Auth extraction.** Identify authentication mechanisms from headers, cookies, and token patterns in the captured requests.
- **Protection detection.** Identify CDN/WAF signatures, CAPTCHA scripts, fingerprinting code, and request signing patterns from the traffic and loaded scripts.

### Output

A **Page Context Document** — a structured profile containing all five categories above. This document is carried forward into every subsequent phase.

## Phase 2 — MAPPING

### Goal

Build a rich sample set for each relevant endpoint across multiple sessions. Understand what's static, what's dynamic, and how the API behaves over time. Produce enough data to confidently generate code later.

### What We Capture Per Endpoint

#### Request Patterns

- Headers: which are always the same vs change between calls
- Query parameters: fixed vs user/context driven
- URL variance: path parameters, IDs, how the URL itself changes
- Request body patterns for POST/PUT calls

#### Response Patterns

- Fields always present vs sometimes missing
- Field value types and ranges (always integer? always positive? bounded?)
- Response size variance — 3 items sometimes, 300 other times
- Different response shapes for the same endpoint (error vs success, empty vs populated)

#### Token & Session Analysis (Across Sessions)

- Tokens that persist across sessions vs tokens that rotate
- Rotation cadence — per-session, per-request, time-based
- Nonces, signatures, cache-busters
- Session lifecycle — how long before expiry

### User Interaction During Mapping

#### 2a — User Captures Across Multiple Sessions

The user repeats browsing the site across separate sessions, exporting a new HAR file each time:

- Session 1: Normal browsing through areas of interest → `session1.har`
- Session 2: Close browser, reopen, browse the same areas again — generates fresh tokens/cookies → `session2.har`
- Session 3: Browse at a different time — captures how data (e.g., odds, event availability) shifts → `session3.har`
- Additional sessions as needed for confidence

The user decides when they've captured enough. More HAR files = higher confidence in the analysis.

#### 2b — Claude Performs Differential Analysis

After each session, Claude compares against previous sessions and identifies patterns automatically from the captured traffic:

- "The `X-Auth-Token` header changed between sessions but stays constant within a session"
- "The odds values shifted between captures but the response structure is identical"
- "There's a `_ts` query parameter that appears to be a cache-buster — changes every request"
- "The event IDs are stable across sessions — we can treat these as persistent identifiers"

#### 2c — Claude Identifies Gaps in Coverage

Based on what's been captured so far, Claude flags areas with low confidence:

- "I've only seen this endpoint return football events — the response shape for other sports is unknown"
- "No error responses captured for this endpoint — error format is unknown"
- "Pagination only observed up to page 2 — cursor pattern is inferred but not fully confirmed"

The user can then decide whether to do additional capture sessions to fill these gaps or proceed with what's available.

### Output

A **Sample Catalog** per endpoint containing:

- Multiple request/response pairs grouped by session
- Annotations on each field: static, dynamic-per-session, dynamic-per-request, user-dependent
- Identified parameters: what the user controls vs what the client must generate
- Confidence level: "seen 12 times, high confidence" vs "seen once, needs more samples"

## Phase 3 — REPORT

### Goal

Synthesize everything from recon and mapping into a human-readable API intelligence report. The user reads this and makes informed decisions before any code is generated.

### Why This Phase Exists

Before jumping into code generation, the user needs to understand what they're working with. This is the decision point — what's feasible, what's going to be painful, what to prioritize. The report is written for the user, not for machines.

### Report Structure

#### Site Overview

High-level summary in plain language. Example: "tipsport.cz is an Angular-based betting platform. Event and odds data is served from `api.tipsport.cz/rest/` via REST endpoints. Live scores are delivered over a SignalR WebSocket on `live.tipsport.cz`. The site sits behind Cloudflare but API calls from an authenticated session are not challenged."

#### Data Source Map

Clear breakdown of what data comes from where, mapped directly to what the user said they wanted in Phase 1:

- Events → which endpoint, what it returns
- Odds → where they come from (embedded in events response? separate endpoint? WebSocket push?)
- Scores → REST endpoint, WebSocket only, or both?

This section answers: "For each piece of data you wanted, here's exactly where it lives."

#### Endpoint Catalog

Each discovered endpoint with:

- Plain-language description of what it does
- Parameters it accepts
- Rough shape of what it returns (not full schema — that's for code generation)
- Confidence rating based on sample count from Phase 2
- Dependencies between endpoints ("you need `eventId` from the events list to call match detail")

#### Auth & Token Summary

Practical assessment of what's needed to make calls work:

- What tokens/cookies are required
- How they're obtained and how they rotate
- Feasibility: can this be replicated programmatically, or is it tied to browser JS execution?

#### Real-time Data Summary

- What's available over WebSocket/SSE
- Subscription model: how you subscribe, what you receive
- Message format and observed frequency

#### Protection & Feasibility Assessment

Honest assessment of what the user is up against:

- What protections are in place and how they affect programmatic access
- What's straightforward to replicate vs what requires workarounds (e.g., headless browser)
- Hard blockers if any exist

#### Behavioral Notes

- Polling intervals, rate limit signals
- Data freshness observations
- Pagination strategy, sorting defaults
- Quirks: "the API returns different field names for football vs hockey events"

#### Gaps & Unknowns

- What we didn't capture or couldn't determine
- Confidence gaps in the sample catalog
- Recommendations for additional mapping if the user wants to fill gaps before proceeding

### Output

An **API Intelligence Report** — a document the user can read, share with their team, or use as a reference. The structured data from recon and mapping still exists underneath for code generation phases.

## Phase 4 — TYPE GENERATION

### Goal

Generate typed class definitions from discovered API endpoints, turning JSON response samples into ready-to-use code in the user's target language.

### MCP Servers Used

- **apiregen-har** — extracts JSON response/request bodies from loaded HAR data
- **quicktype** (remote, `https://mcp.quicktype.io/mcp`) — infers types from JSON samples and generates code

### How It Works

1. User specifies endpoint pattern(s) and target language
2. Claude extracts the best JSON sample from HAR data for each endpoint
3. Claude feeds the JSON to QuickType with a meaningful type name
4. Claude reviews the generated types, flags potential improvements (dates, enums, nullability)
5. User approves, Claude writes the code to disk

### Supported Languages

ruby, javascript, flow, rust, kotlin, dart, python, csharp, go, cpp, java, scala3, typescript, swift, objc, elm, schema, pike, haskell, php

### Slash Commands

| Command | Phase | What it does |
|---------|-------|-------------|
| `/capture` | Setup | Guide through capturing HAR files |
| `/recon` | 1 | Build Page Context Document from traffic |
| `/mapping` | 2 | Cross-session differential analysis |
| `/report` | 3 | Synthesize full API Intelligence Report |
| `/schema` | 3→4 | Infer JSON schema for specific endpoints |
| `/typegen` | 4 | Generate typed classes via QuickType |
| `/investigate` | Any | Ad-hoc deep-dive into specific endpoints |
